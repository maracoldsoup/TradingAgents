# 서비스 MVP 스펙

## 한 줄 정의

국내/해외 종목, ETF, 테마를 모르는 사람도 "왜 움직였고, 무엇으로 구성됐고,
다음에 무엇을 봐야 하는지"를 30초 안에 이해하게 만드는 리서치 콘텐츠 서비스.

자동매매, 수익률 보장, 종목 추천 서비스가 아니다. 현재 MVP는 설명, 시각화,
출처, 사후 검증이 있는 콘텐츠 서비스다.

## 참고 서비스 역할

### AntWiki: 사용자 경험 기준

사용자가 처음 보는 화면은 앤트위키처럼 쉬워야 한다.

- 오늘 뜨는 테마와 종목을 먼저 보여준다.
- "왜 움직였나"가 숫자 대시보드보다 먼저 온다.
- 섹터 -> 테마 -> 종목/ETF로 내려가는 탐색을 제공한다.
- ETF와 테마는 구성 지도를 핵심 화면으로 둔다.
- Bull/Bear 논쟁은 긴 리포트가 아니라 짧은 카드로 압축한다.

### NabakAI: 신뢰 레이어 기준

화면 뒤에는 나박AI처럼 출처와 검증이 있어야 한다.

- 원천 데이터, AI 요약, 자체 계산, 추정을 구분한다.
- DART/SEC/뉴스/시세/ETF holdings 같은 출처를 카드에 붙인다.
- 콘텐츠 발행 후 5일/20일 가격 변화를 기록한다.
- 맞은 근거와 틀린 근거를 숨기지 않는다.
- 국내 이벤트와 해외 이벤트를 서로 연결한다.

### Cloudflare: 운영 콘솔 기준

운영자는 Cloudflare처럼 서비스 상태를 제어해야 한다.

- 수집 워커 상태
- 캐시 상태
- 후보 큐
- 비용 가드
- 발행 가능/불가
- API 상태

Cloudflare 스타일은 공개 서비스 첫 화면이 아니라 `/ops` 운영 콘솔에 적용한다.

## 정보 구조

### 공개 서비스

사용자가 보는 제품이다.

- `/`: 오늘의 테마, 움직인 종목, ETF/테마 카드
- `/search`: 종목명, 티커, ETF, 테마 검색
- `/sectors`: 섹터 목록 (섹터 수, 섹터별 테마/종목/ETF 수) — 앤트위키
  `/themes` 벤치마크, `docs/content_product_spec.md`의 "섹터 레이어" 참고
- `/sectors/{slug}`: 섹터 하위 테마 목록
- `/stocks/{ticker}`: 개별 종목 위키
- `/etfs/{ticker}`: ETF 구성 해설
- `/themes/{slug}`: 테마 밸류체인 + 이 테마를 담은 ETF 목록
- `/learn`: ETF/테마 초보자 개념 카드
- `/review`: 발행 후 5일/20일 사후 점검

`/`, `/sectors`, `/themes/{slug}`는 종목과 ETF를 같은 드릴다운 트리 안에서
같이 보여준다. ETF를 종목과 분리된 별도 탐색 체계로 두지 않는다 — "테마를
아는데 종목은 모르는" 사람이 테마 페이지에서 바로 ETF로 넘어갈 수 있어야
한다.

### 운영 콘솔

운영자가 보는 제품이다.

- `/ops`: 전체 상태
- `/ops/queue`: 후보 큐
- `/ops/sources`: Toss/DART/SEC/ETF holdings/news 수집 상태
- `/ops/publish`: 발행 대기 콘텐츠
- `/ops/cost`: LLM/API 비용 가드
- `/ops/api`: 내부 API 상태

## 공개 화면 우선순위

### 1. 홈

목표: 사용자가 들어오자마자 "오늘 볼 만한 흐름"을 이해한다.

필수 영역:

- 오늘 움직인 테마
- 왜 올랐나/왜 떨어졌나 카드
- 국내/해외 연결 카드
- ETF 구성 카드
- 초보자용 설명 카드

홈에 운영 지표를 노출하지 않는다. 후보 수, 비용 가드, 큐 상태는 `/ops`로 보낸다.

### 2. 종목 페이지

필수 질문:

- 이 회사는 무엇을 하는가?
- 오늘 왜 움직였나?
- 어떤 사업/지역/제품에 노출되어 있나?
- 상승 관점과 주의 관점은 무엇인가?
- 다음에 봐야 할 이벤트는 무엇인가?

필수 시각화:

- 가격 추이
- 거래량 변화
- 사업/지역 구성
- 이벤트 타임라인
- 가격 사다리는 entry/stop/target이 있을 때만 표시

### 3. ETF 페이지

필수 질문:

- 이 ETF는 무엇을 묶은 상품인가?
- 오늘 왜 움직였나?
- 상위 보유 종목은 무엇인가?
- 섹터/국가 비중은 어떤가?
- 특정 종목 쏠림이 있는가?
- 이 ETF는 어떤 섹터/테마를 대표하는 상품인가? (`primary_sector`, 해당
  섹터 페이지로 돌아가는 링크)

필수 시각화:

- 상위 10개 보유 종목 막대
- 섹터 비중
- 국가 비중
- 집중도 경고
- 환율 노출 표시

ETF 구성 데이터가 없으면 발행하지 않는다. `primary_sector`가 없어도 발행은
막지 않는다 — 섹터 링크가 비어 있을 뿐이다.

### 4. 테마 페이지

필수 질문:

- 이 테마는 어떤 산업/사건과 연결되는가?
- 오늘 왜 주목받는가?
- 밸류체인은 어떻게 나뉘는가?
- 국내 대표 종목과 해외 대표 종목은 무엇인가?
- 촉매와 리스크는 무엇인가?
- 종목을 하나씩 고르기 어렵다면, 이 테마를 담은 ETF는 무엇인가?

필수 시각화:

- 밸류체인 지도
- 국내/해외 대표 종목 비교
- 테마 종목 최근 흐름
- 촉매 이벤트 타임라인
- 리스크 매트릭스
- 테마 ETF 목록 (`primary_sector`가 이 테마의 섹터와 일치하는 ETF, 있는
  경우에만 — 발행 게이트 대상 아님)

테마 밸류체인 데이터가 없으면 발행하지 않는다.

## 데이터 계층

### 무료/저비용 우선

- Toss: 국내/미국 종목 정보, 현재가, 캔들, 환율, 장 캘린더
- DART: 국내 공시
- SEC: 미국 공시
- ETF holdings CSV/JSON: 운용사 또는 공개 파일
- 뉴스/RSS: 원인 카드 후보
- 로컬 JSON/CSV: 사람이 정리한 테마 맵

외부 LLM API는 MVP 기본 경로에서 사용하지 않는다.

### LLM 사용 원칙

- 숫자를 만들지 않는다.
- 출처 없는 원인을 단정하지 않는다.
- 로컬/무료 모델은 초안 정리에만 쓴다.
- 유료 모델은 발행 후보가 좁혀진 뒤 카피 품질 개선에만 쓴다.
- 동일 데이터와 날짜의 결과는 캐시한다.

## 발행 기준

공개 서비스에 노출 가능한 최소 조건:

- 한 줄 정의가 있다.
- 왜 움직였나 카드가 있다.
- 원천 데이터 또는 출처가 표시된다.
- ETF는 holdings/sector/country 구성이 있다.
- 테마는 value_chain과 대표 종목이 있다.
- 리스크 또는 데이터 부족 경고가 있다.
- 투자 조언처럼 보이는 문구를 피한다.

## 구현 순서

### 1단계: 스펙 고정

이 문서를 기준으로 공개 서비스와 운영 콘솔을 분리한다.

### 2단계: 데이터 모델

`content_snapshot`을 공개 서비스용 asset 모델로 정리한다.

필수 API:

- `GET /api/assets`
- `GET /api/assets/{id}`
- `GET /api/themes`
- `GET /api/ops/status`

### 3단계: 공개 서비스 화면

먼저 `/`, `/stocks/{ticker}`, `/etfs/{ticker}`, `/themes/{slug}`만 만든다.

### 4단계: 운영 콘솔

후보 큐, 수집 상태, 비용 가드, 발행 상태를 `/ops`로 옮긴다.

### 5단계: 데이터 보강

20개 이상 실제 후보를 채운 뒤에만 paid-model 비교나 장기 검증을 검토한다.

## 지금 중단할 것

- 운영 대시보드를 공개 서비스처럼 포장하지 않는다.
- 후보 큐 숫자를 홈 첫 화면에 내세우지 않는다.
- HTML 미리보기를 서비스라고 부르지 않는다.
- 12개월 검증부터 하지 않는다.
- Gemini 같은 외부 LLM API를 기본 경로에 넣지 않는다.
